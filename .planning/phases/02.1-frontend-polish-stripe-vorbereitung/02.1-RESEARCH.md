## Summary

Phase 02.1 fügt vier UI-Polish-Items, einen Stripe-Webhook-Skeleton ohne Live-Account, vier UAT-Vorbereitungs-Markdowns und WIZ-04 Reload-Resume ein. Die wichtigste technische Erkenntnis dieser Recherche: **die Pipeline läuft heute synchron innerhalb des `/api/wizard/generate`-Request-Lifecycles** (`maxDuration = 300`s), das Frontend hat **null Live-Updates während der 90–150s** und im aktuellen `WizardShell` wird ein Reload während `phase=generating` zwar serverseitig zu Ende laufen, aber das Frontend startet auf der Antrag-Seite OHNE laufende Pipeline-Detection.

**Primary recommendation:** 

1. **Stripe-Webhook:** Kanonisches App-Router-Pattern mit `await req.text()` + `headers()` aus `next/headers` (verifiziert gegen offizielles `stripe-node`-Repo). SDK `stripe@^20.4.1` ist im Repo, in `lib/stripe/client.ts` als Singleton vorhanden. Webhook-Skeleton dispatched 3 Events, Business-Logic als TODO-Stubs.
2. **GeneratingProgress (D-04):** **Polling-Variante** wählen — neuer Stage-State im Pipeline-DB-Row (JSONB-Erweiterung in `antrag_data.generation.stage`), bei jedem `emit()` aktualisiert, Frontend pollt alle 2s über bestehenden `/api/wizard/[token]`-Endpoint. KEIN SSE — passt nicht zum bestehenden Stack (Polling-Pattern ist bereits in `CheckoutSuccessClient.tsx` etabliert, dort 2-s-Tick).
3. **WIZ-04 Reload-Resume (D-12, D-13):** Drei diskrete Lücken. (a) `WizardShell.startGeneration` muss bei vorhandener `phase=generating`-Session NICHT `/api/wizard/generate` neu starten, sondern nur pollen. (b) `/api/wizard/generate` muss idempotent sein — heute crasht es bei zweitem Aufruf in `phase=generating` mit 409. (c) Stage-Persistenz muss überleben (Pipeline schreibt Stage in DB, nicht nur ein in-memory Callback).
4. **UI-Polish 1–3 (D-01/02/03/05):** Konkrete Datei-+-Diff-Vorschläge unten. `react-markdown@^10.1.0` und `remark-gfm@^4.0.1` sind bereits installiert. Sektions-Anker-Navigation kann den bestehenden `MARKDOWN_COMPONENTS`-h2-Renderer um `id`-Attribute erweitern, kein neuer Library-Import.

## 1.2 Kanonisches Webhook-Pattern für Next.js 16 App Router

**Quelle:** Offizielles Stripe-Repo `stripe-node/examples/webhook-signing/nextjs/app/api/webhooks/route.ts`
`[CITED: github.com/stripe/stripe-node/blob/master/examples/webhook-signing/nextjs/app/api/webhooks/route.ts]`

**Kanonisches Skelett (an EduFunds-Patterns angepasst):**

```typescript
// app/api/stripe/webhook/route.ts
import { NextRequest, NextResponse } from "next/server";
import { headers } from "next/headers";
import type Stripe from "stripe";
import { getStripe, stripeConfigured } from "@/lib/stripe/client";
import { markSessionPaid } from "@/lib/wizard/session";

// WICHTIG: KEINE bodyParser-Konfig nötig wie im alten Pages-Router-Pattern.
// App Router liefert per Default Raw-Body via req.text(); Next 16 reicht ihn 1:1 durch.

export async function POST(req: NextRequest) {
  if (!stripeConfigured() || !process.env.STRIPE_WEBHOOK_SECRET) {
    return NextResponse.json(
      { error: "Webhook nicht konfiguriert (STRIPE_WEBHOOK_SECRET fehlt)" },
      { status: 503 }
    );
  }

  const stripeSignature = (await headers()).get("stripe-signature");
  if (!stripeSignature) {
    return NextResponse.json({ error: "stripe-signature header fehlt" }, { status: 400 });
  }

  const rawBody = await req.text(); // RAW! NICHT req.json() — Verification scheitert sonst.

  let event: Stripe.Event;
  try {
    event = getStripe().webhooks.constructEvent(
      rawBody,
      stripeSignature,
      process.env.STRIPE_WEBHOOK_SECRET
    );
  } catch (err) {
    const msg = err instanceof Error ? err.message : "unbekannter Verification-Fehler";
    console.error("[stripe/webhook] Signature-Verification fehlgeschlagen:", msg);
    return NextResponse.json({ error: `Webhook-Verification: ${msg}` }, { status: 400 });
  }

  try {
    switch (event.type) {
      case "checkout.session.completed": {
        const cs = event.data.object as Stripe.Checkout.Session;
        const sessionToken = cs.metadata?.wizard_session_token;
        if (!sessionToken) {
          console.warn("[stripe/webhook] checkout.session.completed ohne wizard_session_token");
          break;
        }
        await markSessionPaid(sessionToken, {
          stripeSessionId: cs.id,
          stripeCustomerEmail: cs.customer_details?.email ?? undefined,
          tier: cs.metadata?.tier,
        });
        // TODO PAY-03 (v2): Invoice-Mail an Käufer (Resend ist bereits installiert: resend@^4.1.1)
        // TODO PAY-04 (v2): Steuer-Konformitäts-Audit-Log
        console.log(`[stripe/webhook] Session ${sessionToken} markiert als paid`);
        break;
      }
      case "checkout.session.expired": {
        const cs = event.data.object as Stripe.Checkout.Session;
        const sessionToken = cs.metadata?.wizard_session_token;
        // TODO PAY-03 (v2): User-Session-State auf "checkout_expired" setzen, UI-Hinweis "Zahlung abgebrochen"
        console.log(`[stripe/webhook] Session ${sessionToken ?? "unknown"} expired`);
        break;
      }
      case "charge.refunded": {
        const ch = event.data.object as Stripe.Charge;
        // TODO PAY-03 (v2): Antrag in "refunded"-Status — paid_token invalidieren?
        // Datenbank-Implikation: ki_antraege.status um "refunded" erweitern (neue Migration v2)
        console.log(`[stripe/webhook] Charge ${ch.id} refunded`);
        break;
      }
      default:
        console.log(`[stripe/webhook] Event-Type ${event.type} nicht behandelt — ignoriert`);
    }
  } catch (err) {
    // Business-Logic-Fehler: 500 zurück, Stripe wird Webhook automatisch retryen.
    console.error(`[stripe/webhook] Dispatch-Fehler für ${event.type}:`, err);
    return NextResponse.json(
      { error: err instanceof Error ? err.message : "Dispatch-Fehler" },
      { status: 500 }
    );
  }

  return NextResponse.json({ received: true });
}
```

**Kritische Pattern-Eigenschaften (alle [CITED]):**

1. **`await req.text()` — NICHT `req.json()`.** JSON-Parsing zerstört das Raw-Format und die Signature-Verification scheitert. Quelle: offizielles Stripe-Doc-Beispiel + Stripe-Webhook-Signature-Doc.
   `[CITED: docs.stripe.com/webhooks/signature]`
2. **`headers()` aus `next/headers`** — in Next.js 16 App Router ist `headers()` jetzt **async** (Breaking Change zwischen Next 14 und 15). Im Beispiel oben: `(await headers()).get("stripe-signature")`.
   `[CITED: github.com/stripe/stripe-node/blob/master/examples/webhook-signing/nextjs/app/api/webhooks/route.ts]`
3. **Kein `export const config = { api: { bodyParser: false } }`-Konstrukt.** Das war Pages-Router. App Router schaltet bodyParser nicht ein — `req.text()` gibt direkt Raw zurück.
   `[CITED: docs.stripe.com/webhooks (App Router pattern)]`
4. **5xx zurückgeben bei Business-Logic-Fehler** = Stripe retryed automatisch. 4xx = Stripe gibt auf. Skeleton sollte NIE 4xx werfen für rein interne Fehler.
   `[CITED: docs.stripe.com/webhooks (Receive Stripe events)]`

## 3.3 Empfohlenes Pattern: Stage-Field in DB + Client-Polling

**Begründung gegen SSE:**
- Aktueller Pattern in `CheckoutSuccessClient.tsx` ist Polling (`POLL_INTERVAL_MS = 2000`, `MAX_WAIT_MS = 60000`). Konsistenz schlägt theoretische Eleganz.
- SSE in Next.js 16 App Router erfordert Streaming-Response (`new Response(stream)`), zusätzliche Reconnect-Logik im Client, kein klares Idempotenz-Modell bei Reload.
- Polling-Latenz (2 s) ist für einen 90–150 s-Lauf vernachlässigbar — User sieht die 7 Stages mit ±2 s Lag.
- Polling ist vereinbar mit Hetzner+Traefik+Docker-Stack (kein Proxy-Buffering, kein `X-Accel-Buffering: no`-Header nötig).

**Begründung gegen "neuer SSE-Endpoint `/api/wizard/[token]/progress`":**
- Doppelte Persistenz-Quelle wäre nötig (SSE liest aus … wovon? Wenn aus DB, dann ist Polling über `/api/wizard/[token]` simpler).
- Mehr Code, mehr Failure-Modes.

**Status-Berechnung in `GeneratingProgress`:**

```typescript
// Reihenfolge muss mit STAGES-Index übereinstimmen
function stageStatus(stageKey: string, currentStage?: string): "pending" | "active" | "done" {
  if (!currentStage) return "pending";
  if (currentStage === "done") return "done";
  const order = ["outline", "section", "critique", "revision", "recheck", "finanzplan", "consistency"];
  const currentIdx = order.indexOf(currentStage);
  const myIdx = order.indexOf(stageKey);
  if (myIdx < 0 || currentIdx < 0) return "pending";
  if (myIdx < currentIdx) return "done";
  if (myIdx === currentIdx) return "active";
  return "pending";
}
```

**Edge-Case `recheck` und `consistency` sind bedingt:** Wenn die Pipeline diese Stages skipped (Findings=0 oder Posten=0), springt `currentStage` direkt zu `finanzplan` bzw. `done`. Status-Logik oben rendert die übersprungenen Stages dann korrekt als "done" (myIdx < currentIdx → "done"). **Acceptable** — User sieht "abgehakt" ohne tatsächlichen LLM-Call, was semantisch stimmt (Stage hat nichts zu tun gehabt).

## Common Pitfalls

### Pitfall 1: `req.json()` statt `req.text()` im Webhook
**Was geht schief:** Stripe-Signature-Verification scheitert mit kryptischem Fehler "No signatures found matching the expected signature". User klickt auf "Test-Mode" und sieht 400er.
**Warum:** JSON-Parsing zerstört das Raw-Format (Whitespace-Normalisierung, Key-Order). Stripe signiert die exakten Bytes.
**Wie verhindern:** **Immer** `await req.text()` als erstes nach Header-Read. Kommentar im Code-Header lassen.
**Warnzeichen:** `SignatureVerificationError` in Logs, `whsec_xxx`-Mismatch wird falsch interpretiert.
`[CITED: docs.stripe.com/webhooks/signature]`

### Pitfall 2: `headers()` ist async in Next.js 16
**Was geht schief:** Code aus Tutorials < 2025: `const sig = headers().get(...)` — TypeError, weil `headers()` jetzt Promise zurückgibt.
**Warum:** Breaking Change Next 14 → 15.
**Wie verhindern:** `(await headers()).get(...)` mit `await`.
**Warnzeichen:** TS-Fehler "Property 'get' does not exist on type 'Promise<ReadonlyHeaders>'".
`[CITED: github.com/stripe/stripe-node/blob/master/examples/webhook-signing/nextjs/app/api/webhooks/route.ts]`

### Pitfall 3: Webhook gibt 4xx bei Business-Logic-Fehler
**Was geht schief:** Pipeline schlägt fehl → 4xx → Stripe denkt "Endpoint kaputt, will nicht retry" → checkout.session.completed verloren.
**Warum:** Stripe retried 5xx (3 Tage Exponential-Backoff), 4xx wird verworfen.
**Wie verhindern:** Bei Verification-Fehler 400 (will man auch nicht retryen — Signature falsch). Bei Dispatch-Fehler in Business-Logic 500. Header-Fehler 400. Konfig-Fehler 503.
**Warnzeichen:** Stripe-Dashboard zeigt Webhook als "Failed", aber kein Retry-Counter steigt.
`[CITED: docs.stripe.com/webhooks (Receive Stripe events)]`

### Pitfall 4: `phase=generating` ohne Stage-Heartbeat = Polling pollt endlos
**Was geht schief:** Pipeline crasht (Container-OOM, 300s-Timeout), Phase bleibt `generating`. Frontend pollt alle 2s, sieht keine Änderung, User starrt 10 Min auf "wird geschrieben".
**Warum:** Kein Liveness-Indikator zwischen Pipeline-Lauf und DB-State.
**Wie verhindern:** Stage-Heartbeat schreibt `generation.stageAt` bei jedem `emit()`. Frontend prüft: `Date.now() - stageAt > 60_000` → "Pipeline-Lauf scheint abgebrochen — neu starten?"-CTA. Min-Timeout-Schwelle 60 s, weil section-Stage bei DigitalPakt 2.0 ≈ 8 Sektionen × 5–10 s = 40–80 s LLM-Calls hat.
**Warnzeichen:** Gleicher Stage-Wert > 60 s.

### Pitfall 5: Sektions-Anker-Slugs kollidieren bei wiederholten h2-Texten
**Was geht schief:** "Vorhaben" als h2 erscheint zweimal (Antrag + Finanzplan-Markdown) → beide bekommen `id="vorhaben"` → Browser scrollt nur zum ersten.
**Warum:** Anker-IDs müssen eindeutig sein.
**Wie verhindern:** Index-Suffix bei Duplikaten: `vorhaben`, `vorhaben-2`. ODER: Markdown-Sektions-Nav nur über Antragsteil generieren, nicht über Finanzplan (Finanzplan hat eh eigene Tabellen-Struktur).
**Warnzeichen:** Klick auf zweiten Sektions-Link springt zum ersten Eintrag.
`[ASSUMED]` Heutige Antrag-Markdown produziert keine Duplikate — aus pipeline.ts:`renderDraft` ist sichtbar, dass jeder Abschnitt unique ist. Aber Plan-Phase sollte trotzdem Test mit Duplikat-Test-Case in Wave 1 ergänzen.

### Pitfall 6: Polling startet doppelt bei React-StrictMode
**Was geht schief:** In Dev-Mode rendert React jeden useEffect zweimal → zwei Polling-Loops parallel → doppelte fetch/s, doppelter State-Update.
**Warum:** StrictMode-Doppel-Mount.
**Wie verhindern:** `cancelled`-Flag in Cleanup-Funktion (Pattern in `CheckoutSuccessClient.tsx` korrekt umgesetzt — als Vorlage nutzen).
**Warnzeichen:** Network-Tab zeigt 2 Requests/2s statt 1 Request/2s.
`[VERIFIED: components/Wizard/CheckoutSuccessClient.tsx Z. 30–80 — Cancel-Pattern]`

## Security Domain

> `security_enforcement` ist nicht explizit in `.planning/config.json` gesetzt — Default-Annahme: enabled.

### Applicable ASVS Categories

| ASVS Category | Applies | Standard Control |
|---------------|---------|-----------------|
| V2 Authentication | nein (Phase 02.1 hat keine User-Accounts; Auth bleibt v2 AUTH-01) | n/a |
| V3 Session Management | partial (Anonyme Wizard-Sessions via session_token UUID + localStorage) | UUIDv4 via `randomUUID()` (Crypto-strong) — bestehend in `session.ts:49` |
| V4 Access Control | partial (paid_token-Gate für Download) | Token-only-Auth, keine User-Roles — bestehend in `getSessionByPaidToken` |
| V5 Input Validation | yes | Webhook-Body-Validation via `stripe.webhooks.constructEvent` (Signature) + JSON-Schema-Cast für Stripe.Event-Type |
| V6 Cryptography | yes | Stripe-SDK macht HMAC-SHA256 für Webhook-Verification (constant-time Compare); UUID-Generation via Node-Crypto |

### Known Threat Patterns für Stripe-Webhooks

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Webhook-Replay (alte Signatur, neuer Body) | Tampering | Stripe-SDK prüft Timestamp im Signature-Header (Tolerance default 5 Min) — eingebaut in `constructEvent` |
| Webhook-Spoofing (falsche Signatur) | Spoofing | `constructEvent` wirft `SignatureVerificationError` — wir antworten 400 |
| Race-Condition `markSessionPaid` (zweimal Webhook) | Tampering | `markSessionPaid` ist idempotent (s. session.ts:151–153 — bestehender Check `if (existing.paidToken) return existing`) |
| Webhook-Endpoint-Enumeration (Angreifer findet `/api/stripe/webhook` und probiert Payloads) | Information Disclosure | Verification-Fehler 400 ohne Detail-Leak; Logs nicht zu Frontend |
| Stage-Heartbeat-Race (zwei Pipeline-Läufe schreiben parallel) | Tampering | `jsonb_set` ist atomic in Postgres; bei zwei parallelen Generates wäre der "letzte gewinnt"-Pattern OK (kein Phantom-State) |
| localStorage-Token-Diebstahl | Information Disclosure | Out-of-Scope für 02.1 — Anonyme Sessions sind explizit Design-Choice (PROJECT.md Out of Scope: "Auth bleibt v2") |

**Concrete: keine NEUEN Threat-Patterns durch Phase 02.1 — alle Mitigations sind in bestehenden Stripe-SDK + `markSessionPaid`-Idempotenz abgedeckt.**
