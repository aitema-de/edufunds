<decisions>
## Implementation Decisions

### UI-Polish-Scope

- **D-01:** Vier konkrete Polish-Items im Scope (alles weitere = Scope-Creep, in Deferred verschoben):
  1. Antrag-Detail-Seite Lese-UX (`AntragResult.tsx` + `app/antrag/[id]/`)
  2. Empty-States in `MyAntraegeClient.tsx` (`/antrag/meine` leer) und Matcher-0-Treffer-Fall
  3. Mobile-Breakpoints für Critical-Paths
  4. Loading-State während 90-150s Pipeline-Generation (`GeneratingProgress.tsx`)
- **D-02:** Antrag-Detail-Lesemodus = In-App-Markdown-Render mit Sektions-Anker-Navigation + Edit-Button pro Sektion. PDF-Download bleibt als Beigabe für Behörden-Einreichung. Begründung: UAT-Anwender liest meist im Browser, nicht im PDF.
- **D-03:** Mobile = Touch-Optimierung der Critical-Paths (Anliegen-Eingabe, Match-Auswahl, Wizard-Antwort-Eingabe, Antrag-Lesen) auf iPhone-Safari + Android-Chrome verifiziert. Tailwind-Default-Breakpoints sind Ausgangspunkt — Researcher/Planer prüft, wo `sm:`/`md:`-Klassen gezielt verfeinert werden müssen.
- **D-04:** GeneratingProgress = 7 Pipeline-Stages aus `lib/wizard/pipeline.ts` live anzeigen (Outline → Section → Critique → Recheck → Revision → Finanzplan → Konsistenz). UAT 28.04. zeigte: User glaubt App ist tot ohne Stage-Hinweise. Kein Estimated-Time, sondern Stage-Tracking.
- **D-05:** Empty-States Action-orientiert mit konkreten CTAs:
  - `/antrag/meine` leer → "Noch kein Antrag begonnen. [Anliegen schildern]"-Button
  - Matcher-0-Treffer → "Kein Programm passt — Anliegen anders formulieren?" mit 2-3 konkreten Reformulierungs-Tipps. Phase-2-D-12 (Match-Empty-State unverändert) wird damit überschrieben — Phase-2-D-12 sagte "heute belassen", 02.1 hebt es auf Action-orientiert.

### Stripe-Code-Vorbereitung

- **D-06:** Stripe-Tiefe minimal: nur Webhook-Skeleton in 02.1. PAY-03 (`invoice_creation`) und PAY-04 (`automatic_tax`) bleiben in `REQUIREMENTS.md` v2 / Out of Scope. Dev-Mock-Paywall (`NEXT_PUBLIC_PAYWALL_DEV_MOCK=1`) bleibt für interne UATs.
- **D-07:** Webhook-Handler als `app/api/stripe/webhook/route.ts` neu anlegen. Signature-Verification über Stripe-SDK-Helper, Mock-Webhook-Secret in Dev. Event-Dispatch für `checkout.session.completed`, `checkout.session.expired`, `charge.refunded` (analog zu `STRIPE_SETUP.md` §3). Handler-Logik darf TODO-Stubs für Business-Logic enthalten (z. B. `// TODO PAY-03: Invoice-Mail an Käufer`) — Skeleton liefert Routing + Verification, nicht End-to-End-Business-Logic.
- **D-08:** Verifikation des Code-Pfads über Stripe-CLI Test-Mode:
  - `stripe listen --forward-to localhost:3101/api/stripe/webhook`
  - `stripe trigger checkout.session.completed` etc.
  - Mit Test-Webhook-Secret in `.env.local`
  - Funktioniert ohne KYC, kostenlos. Smoke-Skript dokumentiert (analog `scripts/smoke-pipeline-with-extractor.ts`).

### Live-UAT-Vorbereitung

- **D-09:** Vier Vorbereitungs-Deliverables in 02.1 — alle Plan/Doku, keine echten Sessions:
  1. UAT-Plan-Template (`.planning/uat/UAT-PLAN-TEMPLATE.md`): Begrüßungs-Skript, Critical-Paths-Beobachtungsliste, Erfolgs-/Abbruch-Kriterien, Datenschutz-Hinweis-Block
  2. Befunde-Tracker-Template (`.planning/uat/UAT-BEFUNDE-TEMPLATE.md`): pro Session ein Memo nach 28.04.-Format (Bug-ID, Severity, Pipeline-Stage, Fix-Vorschlag)
  3. Pilot-Liste (`.planning/uat/PILOTEN.md`): 5-7 Kandidaten aus Koljas Netzwerk (Schule/Schulförderverein, Kontakt, warum geeignet) — als Plan-Checkpoint, Kolja trägt Namen ein, kein selbst-recherchierter Inhalt
  4. Onboarding-Mail-Vorlage (`.planning/uat/UAT-ANSCHREIBEN.md`): Deutsche Pilot-Mail (Was EduFunds, was UAT-Session bedeutet, 45 Min, Screen-Sharing, Datenschutz, Termin-Vorschlag)
- **D-10:** UAT-01/-02/-03 (echte Sessions) bleiben in Phase 6. 02.1 liefert nur die Templates und die Pilot-Liste als Phase-6-Startpunkt.

### WIZ-04 Reload-Resume (von Phase 5 gezogen)

- **D-11:** WIZ-04 wird von Phase 5 nach 02.1 verschoben. Begründung: Live-UAT in Phase 6 mit broken Reload-Resume ist Pilot-Session-Killer ("Tab versehentlich geschlossen, alles weg"). Phase 5 schrumpft auf WIZ-01/02/03 (reines Pipeline-Tuning). Saubere Trennung Markt-Reife (02.1) vs. Pipeline-Quality (Phase 5).
- **D-12:** Volle WIZ-04-Spec gemäß ROADMAP Phase-5 Success Criteria #5: User kann Wizard-Tab schließen oder Hard-Reload während `phase=interviewing` UND `phase=generating` ausführen und über `/antrag/meine` ODER direkte programmspezifische Wizard-URL exakt an letzter Frage/Phase fortsetzen. Keine reduzierte Variante.
- **D-13:** Implementation nutzt bestehende `session_token`-Persistenz (`lib/wizard/session.ts`, Migration `db/migrations/002_wizard_session.sql`). Researcher prüft: was fehlt heute am Resume-Pfad — Generating-Phase wird vermutlich nicht idempotent fortgesetzt (Pipeline-Lauf läuft im Backend weiter, Frontend muss bei Reload den laufenden/abgeschlossenen Lauf erkennen statt neu starten).

### Resolutions nach RESEARCH.md (2026-05-06)

- **D-14:** Edit-Button auf Antrag-Detail-Seite (D-02 Verfeinerung) verlinkt auf den bestehenden Wizard-Edit-Pfad (`/antrag/[id]/wizard/edit-...` oder die in Bug-Fix 30.04. eingeführte `edit-answer`-Route). Kein neuer "Sektion-regenerieren"-Endpoint. Begründung: nutzt bestehende Pipeline-Edit-Logik, kein neuer Code-Pfad.
- **D-15:** `failed`-Phase-UI bekommt einen Mini-Fix in 02.1: Wenn Pipeline `phase=failed` setzt, rendert Frontend eine Fehler-Card mit "Erneut versuchen"-CTA statt heutigem Bug (kein User-Feedback). Wird im Plan rund um D-04 (GeneratingProgress) verankert. Aufwand klein, schließt bekannte Lücke vor Live-UAT.

### Claude's Discretion

- Genaue Stage-Beschriftungen in `GeneratingProgress` (deutsche Texte für 7 Stages — Researcher schlägt vor, Kolja kann editieren)
- Mobile-Breakpoint-Schwellen (`sm:`/`md:`/`lg:`) — Tailwind-Default-Werte, ggf. Custom-Werte wenn Critical-Path-Test es zeigt
- PDF-Download-Layout (Schriftart, Logo, Margins) — Standard-Generation reicht, Polish nicht in 02.1
- Empty-State-Illustrationen / Icons — keine custom Illustrationen, Heroicons + Tailwind reichen
- Webhook-TODO-Stub-Granularität (welche Business-Logic-Hooks vorgesehen sind)
- Pilot-Liste-Datenfelder (welche Spalten für Kandidaten-Tracking)
